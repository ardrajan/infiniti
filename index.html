mCLASS:CIRCLE is an early childhood assessment software in English and Spanish for pre-kindergarten.

To comply with my non-disclosure agreement, I have omitted or obfuscated confidential information in this case study. Some of the information presented in this case study is my own, and does not necessarily reflect the views of Amplify.

Business Requirement

The assessments available under mCLASS:CIRCLE were traditionally administered using paper and pencil, and the data was analyzed manually. The original mCLASS:CIRCLE application was available in two parts. Individual assessment of students was administered using a standalone application that ran on a number of devices running the Microsoft Windows OS. Results were synced, and aggregate reporting and instructional next steps were available through a web portal, optimized for desktop browsers.

In order to stay relevant in the market, Amplify wanted to port the application to iOS and Android tablets.

View the old application

My Role and Process

I was the lead UX designer for this project, and led a team that comprised of a visual designer and user researcher. Here is the design process that I followed for this project in brief:

  • User research

  • Design workshops with internal stakeholders

  • Documentation

  • Wireframing and prototyping

  • RITE (Rapid Iterative Testing and Evaluation) testing of wireframes and prototypes

  • Visual design

  • Support front-end effort

The Design Challenge

As mentioned before, the software solution provided two different experiences – a standalone (offline) application for assessment, and a web portal for analysis. This dual system was proving to be cumbersome for teachers because working with students’ data required the teacher to be necessarily online.

Secondly, some of the assessments required the student to respond to content presented on the screen. Each response of the student was required to be discreetly marked as correct, incorrect or skipped. On devices that had keyboards, the teacher could use the keyboard to record students' responses in the standalone application. However, since touch tablets do not ship with a keyboard, the mechanism for discreetly recording students’ responses needed to be redesigned.

In short, we had to re-think the entire system of assessment and analysis while making it relevant to touch devices.

User Research

Since this product had been the market for a while, the most obvious place to start was talking to teachers who were already using the product, and Amplify trainers who were conducting training sessions on how to use the product.

Goals for User Research

  • Observe and understand how users were using the existing product.

  • Discover pain-points with the existing product.

  • Discover what was working well with the existing product.

Design Workshops

Once the user research was complete, the design team synthesized the findings. I setup a series of workshops with internal stakeholders. The workshops were conducted over a duration of three weeks, and included cross-functional team members, including former teachers.

Goals for the Wokshops

  • Define the overall product vision.

  • Confirm users for the product.

  • Collaboratively brainstorm ideas for meeting the design challenge.

  • Define the MVP.

  • Identify potential ideas that need user testing.

The Approach

In order to make the workshop fun and engaging, I created a theme around DC Comics’ Superhero Universe. There was an underlying story, and each workshop activity carried forward the story arc while introducing participants to the users, user challenges, and calls to action for generating the product vision, user scenarios, and design ideas collaboratively.

"Mobility and time are of paramount importance since a lot of children at this age have short attention spans. We need to maximize instructional time for students."

Documentation

Personas

Based on the user research and design workshops, I worked with the user researcher to create two primary personas. The first was a 25-30 year old pre-K teacher who was very comfortable with technology. The second persona was a 55-60 year old teacher who needed constant training in the usage of technology in the classroom. For both personas,

  • Saving time on assessments was important so that the saved time could be utilized for instructional time.

  • Student data was very important, and the data had to be easily understandable and actionable.

User Flows

I created user flow diagrams to streamline the assessment and analysis flows. These user flows were constantly refined based on feedback from pre-K teachers during user testing sessions.

Wireframing and Prototyping

I developed a series of wireframes for the application using Photoshop.

Since a large number of features were being designed for the touch platform, I used Justinmind to create prototypes. This helped a lot since we could put the prototype on actual devices and test it out with users.

RITE (Rapid Iterative Testing and Evaluation) Testing

We used RITE testing to test the prototypes with pre-K teachers. The process involved testing prototypes with 2-3 users, and immediately making changes to the designs based on user feedback. I worked with the user researcher to come up with the goals and questions for each round of testing.

For each test session, I was present along with the user researcher and product owner. In addition, two developers from the team were invited for each session so that they could directly hear feedback from teachers. Involving team members with the user testing sessions established empathy for the user across the team.

The RITE testing method was very useful in iterating designs quickly and arriving at user-centric solutions with great experiences.

Visual Design

Once we had a clean user-validated design experience, I worked with the visual designer to finalize the look and feel for the application. We presented a couple of visual design options to our internal stakeholders before finalizing the visual design. Towards the end of our user testing session, we presented prototypes of user features with actual visual designs to see how users reacted. Their feedback was incorporated into the final visual designs.

Supporting Front-End Efforts

In order to fast track front-end development within Amplify, we have developed an internal front-end library. I have been an active contributor to this library since its inception.

For mCLASS:CIRCLE, I worked with the development team in two-week sprints to create UI widgets for the front-end library, and front-end implementation for product features. In some cases, I also wrote Selenium tests for testing the front-end code for some of the features.

The Outcome

mCLASS:CIRCLE successfully launched for the school year 2014-2015 across multiple states in the US.
